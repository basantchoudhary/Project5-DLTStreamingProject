Metadata Tables Design

Five control tables + a run log drive the whole framework. Onboarding a table = insert rows, no code.

Design principle: masters hold dimensions, config holds key/value

Fixed, queried-on columns (ids, table_group_no, enabled) live on the master tables so the loader can filter cheaply. Everything else — the "how to load" knobs — lives as flexible key/value rows so new settings never need a schema change.

Global knobs: target_catalog, scd1_schema, scd2_schema, checkpoint_root, schema_root, qlik_op_col, qlik_delete_code, liquid_cluster_auto.

How meta_loader reads them

load_dataset_metadata(spark, source="ORA_SALES", table_group_no=1, dataset_list="ALL")
  → { "ORA_SALES.ORDERS": { dataset_id, source_table, target_table, table_group_no,
                            primary_keys, sequence_by, scd_type, load_type, cluster_by },
      "ORA_SALES.CUSTOMERS": { ... } }
# master columns overlaid with all of that dataset's config key/values, one flat dict.

Onboarding a new table = 2 inserts

INSERT INTO dataset_master VALUES ('ORA_SALES.RETURNS','ORA_SALES','SALES','RETURNS','returns',3,true);
INSERT INTO dataset_config VALUES
  ('ORA_SALES.RETURNS','primary_keys','return_id'),
  ('ORA_SALES.RETURNS','scd_type','both'),
  ('ORA_SALES.RETURNS','load_type','full');
-- next run of pipeline group 3 picks it up. No code change.
