Low-Level Design

The controller stack and the per-table DLT graph — reader → append_flow → processor → writer.

Controller stack

Per-table DLT graph

For each table the single-table controller defines these nodes. full and ct both append_flow into one _raw streaming table; a processor view standardizes the CDC shape; the writer runs apply_changes into SCD1/SCD2.

Why a _raw + append_flow, not read both directly?

  • One apply_changes target needs one logical source. append_flow lets full and ct feed the same _raw without a union that would break streaming.
  • full can be switched off after seeding (load_type=incr) — drop the flow, keep the table and its state.
  • apply_changes de-dups by keys + _seq, so a full/ct overlap during seeding is harmless.
build_single_table(dataset):
    derived = build_derived_config(dataset, source, platform)
    dlt.create_streaming_table(derived.raw_table)
    if derived.include_full_flow:
        @dlt.append_flow(target=raw)  def full(): reader(full_path)
    @dlt.append_flow(target=raw)      def ct():   reader(ct_path)
    @dlt.view(processed)              def _():    standardize_cdc(read_stream(raw))
    write_scd_targets(derived, processed)   # apply_changes → scd1 / scd2
