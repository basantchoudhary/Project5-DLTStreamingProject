High-Level Flow

From an operational app all the way to query-ready SCD1 / SCD2 tables in Unity Catalog.

Stage by stage

StageWhat happensWhy
SomeApp → OracleThe application commits transactions; Oracle records every change in its redo logs.The redo log is a complete, ordered record of change — the reliable CDC source.
Qlik Replicate (Azure VM)Reads/copies the redo log, extracts changes, and writes snappy-compressed Parquet to ADLS.Turns database-internal change events into cheap, columnar files on cloud storage — no load on the source.
ADLS landingPer table, two folders: <table>__full (snapshot / seed) and <table>__ct (ongoing change table).Full seeds the target once; ct streams every insert/update/delete after.
20 DLT pipelinesEach owns one table_group_no (~10 tables). Auto Loader ingests the files; apply_changes merges CDC.Fan-out = parallelism + blast-radius isolation (one group failing ≠ all 200 tables down).
Unity CatalogEach table lands as a streaming SCD1 and/or SCD2 table with liquid clustering.Query-ready bronze: latest state (SCD1) and full history (SCD2).

Key properties

  • Metadata-driven — onboarding a table is a config row, not code (see metadata design).
  • Log-based CDC — reading redo logs means near-zero impact on the operational database.
  • Exactly-once ingest — Auto Loader checkpoints which files it has seen; re-runs don't double-count.
  • Full → incr lifecycle — seed with __full, then run on __ct only; apply_changes makes the overlap safe.
